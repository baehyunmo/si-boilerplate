## 1. 시스템 컨텍스트 다이어그램 (C4 Model — Level 1)

시스템의 외부 경계와 외부 시스템/사용자와의 관계를 정의한다.

```
                        ┌──────────────┐
                        │   사용자     │
                        │ (Web/Mobile) │
                        └──────┬───────┘
                               │ HTTPS
                               ▼
                    ┌─────────────────────┐
                    │                     │
                    │    [시스템명]        │
                    │   SI 대상 시스템     │
                    │                     │
                    └──┬──────────────┬───┘
                       │              │
              ┌────────▼──────┐  ┌────▼────────────┐
              │  내부 시스템   │  │   외부 시스템    │
              │  (ERP/HR/등)  │  │  (공공API/PG/등) │
              └───────────────┘  └─────────────────┘
```

### 외부 인터페이스 목록

| 대상 시스템 | 연동 방향 | 프로토콜 | 데이터 | 빈도 |

| [시스템명] | Inbound / Outbound / 양방향 | REST / MQ / gRPC | [데이터 설명] | 실시간 / 배치 |

## 4. 인프라 아키텍처

### 4.1 인프라 구성도

```
┌─────────────── 운영 환경 (Production) ───────────────────────┐
│                                                               │
│  ┌─── AZ-A ───────────┐    ┌─── AZ-B ───────────┐          │
│  │ ┌────────────────┐  │    │ ┌────────────────┐  │          │
│  │ │  K8s Worker    │  │    │ │  K8s Worker    │  │          │
│  │ │  Node Pool     │  │    │ │  Node Pool     │  │          │
│  │ └────────────────┘  │    │ └────────────────┘  │          │
│  │ ┌────────────────┐  │    │ ┌────────────────┐  │          │
│  │ │   DB Primary   │  │    │ │   DB Standby   │  │          │
│  │ └────────────────┘  │    │ └────────────────┘  │          │
│  └─────────────────────┘    └─────────────────────┘          │
│                                                               │
│  ┌─── 공통 ──────────────────────────────────────────┐       │
│  │  Load Balancer │ WAF │ CDN │ Object Storage       │       │
│  └───────────────────────────────────────────────────┘       │
└───────────────────────────────────────────────────────────────┘
```

### 4.2 환경별 인프라 사양

| 환경 | K8s 노드 | CPU (총) | Memory (총) | Storage | DB 사양 |
|------|----------|----------|-------------|---------|---------|

| staging | 3 | 12 vCPU | 32 GB | 200 GB | 4 vCPU / 16 GB |
| production | 6 (3+3 HA) | 48 vCPU | 128 GB | 1 TB | 8 vCPU / 32 GB HA |

### 4.3 네트워크 설계

| 네트워크 세그먼트 | 대역 | 용도 | 접근 제어 |
|-------------------|------|------|-----------|

## 5. 데이터 아키텍처

### 5.1 데이터 저장소 구성

| 저장소 | 유형 | 기술 | 용도 | 데이터 특성 |
|--------|------|------|------|-------------|
| Primary DB | RDBMS | PostgreSQL | 트랜잭션 데이터 | 정합성 필수, ACID |
| Document DB | NoSQL | MongoDB | 비정형 데이터 | 스키마 유연성 |
| Cache | In-Memory | Redis | 세션, 캐시 | 고속 접근, TTL |
| Search Engine | 검색 엔진 | Elasticsearch | 전문 검색, 로그 | 비정규화 데이터 |
| Object Storage | 오브젝트 | S3/MinIO | 파일, 이미지 | 대용량 비정형 |

### 5.2 데이터 흐름

```
[사용자 입력] → [API Service] → [Primary DB]
                     │
                     ├──▶ [Cache] (조회 캐시)
                     │
                     ├──▶ [Message Broker] → [Consumer Service] → [Search Index]
                     │
                     └──▶ [Object Storage] (파일 업로드)
```

### 5.3 데이터 백업 및 복구

| 항목 | 정책 | 상세 |
|------|------|------|
| 전체 백업 | 일 1회 | 매일 02:00 UTC+9, 30일 보관 |
| 증분 백업 | WAL 아카이빙 | 실시간, 7일 보관 |
| RPO | 1시간 이내 | WAL 기반 PITR |
| RTO | 2시간 이내 | 자동 복구 스크립트 |
| 백업 검증 | 월 1회 | 복원 테스트 수행 |

### 5.4 데이터 보관 및 파기

| 데이터 유형 | 보관 기간 | 파기 방법 | 법적 근거 |
|-------------|-----------|-----------|-----------|
| 거래 데이터 | 5년 | 영구 삭제 | 전자상거래법 |
| 접속 로그 | 1년 | 자동 삭제 | 정보통신망법 |

## 6. 통합 아키텍처 (Integration Architecture)

### 6.1 API 통합

| 연동 대상 | 방향 | 방식 | 프로토콜 | 인증 | SLA |
|-----------|------|------|----------|------|-----|
| [내부 시스템 A] | Outbound | 동기 | REST (HTTPS) | mTLS | 99.9% |
| [외부 시스템 B] | Inbound | 비동기 | Webhook | API Key | 99.5% |
| [파트너 시스템 C] | 양방향 | 동기+비동기 | gRPC + Kafka | OAuth 2.0 | 99.9% |

### 6.2 메시징/이벤트 아키텍처

```
┌──────────┐    publish     ┌────────────┐    consume    ┌──────────┐
│ Producer │ ──────────────▶│   Kafka    │──────────────▶│ Consumer │
│ Service  │                │  Cluster   │               │ Service  │
└──────────┘                │            │               └──────────┘
                            │ Topics:    │
                            │ - order.*  │
                            │ - user.*   │
                            │ - noti.*   │
                            └────────────┘
```

### 6.3 이벤트 토픽 정의

| 토픽 | 발행자 | 구독자 | 이벤트 | 보관 기간 |
|------|--------|--------|--------|-----------|
| `order.created` | Order Service | Notification, Inventory | 주문 생성 | 7일 |
| `order.completed` | Order Service | Billing, Analytics | 주문 완료 | 7일 |
| `user.registered` | User Service | Notification, CRM | 회원 가입 | 3일 |

### 6.4 연동 오류 처리

| 전략 | 적용 대상 | 구현 방식 |
|------|-----------|-----------|
| 재시도 (Retry) | 일시적 오류 | 지수 백오프 (최대 3회) |
| 서킷 브레이커 | 외부 시스템 호출 | Resilience4j / Istio |
| 데드 레터 큐 (DLQ) | 메시지 처리 실패 | Kafka DLT Topic |
| 보상 트랜잭션 | 분산 트랜잭션 실패 | Saga 패턴 |

## 7. 보안 아키텍처

### 7.1 보안 계층

```
┌─── 경계 보안 ───────────────────────────────┐
│  WAF │ DDoS Protection │ Rate Limiting       │
├─── 네트워크 보안 ───────────────────────────┤
│  Network Policy │ mTLS │ 세그먼테이션        │
├─── 애플리케이션 보안 ───────────────────────┤
│  인증(OIDC) │ 인가(RBAC) │ 입력 검증        │
├─── 데이터 보안 ─────────────────────────────┤
│  암호화(AES-256) │ 마스킹 │ 접근 제어        │
├─── 컨테이너/런타임 보안 ────────────────────┤
│  이미지 스캔 │ Pod Security │ Falco          │
└─────────────────────────────────────────────┘
```

### 7.2 인증/인가 아키텍처

| 구성 요소 | 기술 | 역할 |
|-----------|------|------|

| API Gateway | Kong / Envoy | 토큰 검증, 라우팅 |
| Service Mesh | Istio | mTLS, 서비스 간 인가 |

| 관리자 | CRUD | CRUD | CRUD | R |
| 시스템 관리자 | CRUD | CRUD | CRUD | CRUD |

## 8. 운영 아키텍처

### 8.1 관측성 (Observability)

| 영역 | 도구 | 용도 | 보관 기간 |
|------|------|------|-----------|
| 메트릭 | Prometheus + Grafana | 시스템/애플리케이션 메트릭 | 90일 |
| 로깅 | Loki / EFK | 구조화 로그 수집 및 검색 | 30일 |
| 트레이싱 | Tempo / Jaeger | 분산 트레이싱 | 7일 |
| 알림 | AlertManager + PagerDuty | 장애 알림 | - |

### 8.2 알림 정책

| 구분 | 조건 | 심각도 | 알림 채널 | 응답 시간 |
|------|------|--------|-----------|-----------|
| 서비스 다운 | 헬스체크 3회 연속 실패 | Critical | PagerDuty + Slack | 즉시 |
| 높은 에러율 | 5xx 비율 > 5% (5분간) | Warning | Slack | 15분 이내 |
| 리소스 부족 | CPU/Memory > 80% | Warning | Slack | 30분 이내 |
| 디스크 부족 | 디스크 사용률 > 85% | Warning | Slack + Email | 1시간 이내 |
| 인증서 만료 | 만료일 30일 이전 | Info | Email | 업무 시간 내 |

### 8.3 배포 전략

| 환경 | 배포 방식 | 롤백 방법 | 자동화 수준 |
|------|-----------|-----------|-------------|
| dev | Direct Deploy | Git Revert | 완전 자동 (Push 트리거) |
| staging | Blue-Green | 트래픽 전환 | 완전 자동 (PR Merge 트리거) |
| production | Canary (10% → 50% → 100%) | 자동 롤백 (에러율 기반) | 반자동 (수동 승인 후 자동 배포) |

### 8.4 재해 복구 (DR)

| 항목 | 전략 | 상세 |
|------|------|------|
| DR 사이트 | Active-Standby | 별도 AZ에 Standby 환경 |
| 데이터 복제 | 비동기 복제 | DB 스트리밍 복제 (lag < 1초) |
| 전환 시간 | 30분 이내 | DNS 전환 + 서비스 기동 |
| 전환 조건 | Primary 사이트 장애 지속 15분 이상 | 자동 감지, 수동 전환 |
| DR 훈련 | 반기 1회 | 전환/복귀 테스트 수행 |

> **작성 가이드**
> 1. C4 다이어그램은 Level 1(컨텍스트), Level 2(컨테이너) 수준까지 본 문서에 포함한다.
> 2. Level 3(컴포넌트) 이하는 각 서비스별 상세 설계 문서에서 관리한다.
> 3. 기술 선택은 반드시 ADR로 기록하고, 대안과 선정 사유를 명시한다.
> 4. 보안 아키텍처는 ISMS-P 인증 기준을 참고하여 작성한다.
> 5. 인프라 사양은 부하 테스트 결과에 따라 조정한다.
