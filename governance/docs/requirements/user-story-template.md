## 1. Epic 정의

Epic은 하나의 비즈니스 목표를 달성하기 위한 대규모 작업 단위이다.

### Epic 템플릿

```
Epic ID   : EP-XXX
Epic 명   : [Epic 이름]
목표      : [이 Epic이 달성하고자 하는 비즈니스 목표]
KPI       : [성공 측정 지표]
담당팀    : [담당 팀/에이전트]
예상 기간 : [시작일] ~ [종료일]
상태      : 백로그 / 진행중 / 완료 / 보류
```

---

## 2. Feature 정의

Feature는 Epic을 구성하는 기능 단위이다. 하나의 Epic은 여러 Feature로 분해된다.

### Feature 템플릿

```
Feature ID : FT-XXX
Feature 명 : [Feature 이름]
상위 Epic  : EP-XXX
설명       : [Feature에 대한 간략한 설명]
수용 기준  : [Feature 수준의 완료 조건]
상태       : 백로그 / 진행중 / 완료 / 보류
```

---

## 3. User Story 정의

### 3.1 스토리 작성 형식

```
Story ID     : ST-XXXX
상위 Feature : FT-XXX
스토리       :

  As a [역할/페르소나],
  I want [수행하고자 하는 기능],
  so that [얻고자 하는 비즈니스 가치].

우선순위     : Critical / High / Medium / Low
스토리 포인트: [1 / 2 / 3 / 5 / 8 / 13]
스프린트     : Sprint [N]
담당자       : [담당 에이전트/개발자]
상태         : 백로그 / 진행중 / 리뷰 / 완료
```

### 3.2 Acceptance Criteria (수용 기준)

Given-When-Then 형식으로 작성한다. 하나의 스토리에 여러 시나리오를 정의할 수 있다.

```
AC-1: [시나리오 제목]
  Given [사전 조건 — 시스템의 초기 상태]
  When  [동작 — 사용자가 수행하는 행위]
  Then  [결과 — 기대되는 시스템 반응]

AC-2: [시나리오 제목]
  Given [사전 조건]
  When  [동작]
  Then  [결과]

AC-3 (예외): [예외 시나리오 제목]
  Given [사전 조건]
  When  [비정상적 동작 또는 경계 조건]
  Then  [에러 처리 또는 예외 반응]
```

## 6. Definition of Done (완료 기준)

스토리가 "완료" 상태가 되기 위한 공통 기준:

- [ ] 모든 Acceptance Criteria 충족
- [ ] 단위 테스트 작성 및 통과 (커버리지 80% 이상)
- [ ] 통합 테스트 통과
- [ ] 코드 리뷰 완료
- [ ] API 문서 갱신 (해당 시)
- [ ] 보안 스캔 통과 (Critical/High 0건)
- [ ] 성능 기준 충족 확인
- [ ] 스테이징 환경 배포 및 검증 완료

---

## 7. 우선순위 결정 기준 (MoSCoW)

| 등급 | 설명 | 기준 |
|------|------|------|
| **Critical** | 반드시 해당 스프린트에 포함 | 시스템 운영 불가능 시 |
| **High** | 가급적 해당 스프린트에 포함 | 핵심 업무 기능 |
| **Medium** | 다음 스프린트로 이동 가능 | 업무 편의 기능 |
| **Low** | 백로그에 유지 | 있으면 좋은 기능 |

---

> **작성 가이드**
> 1. 사용자 스토리는 비즈니스 가치(so that) 관점에서 작성한다.
> 2. 하나의 스토리는 하나의 스프린트 내에서 완료 가능한 크기여야 한다.
> 3. 13 포인트 이상의 스토리는 반드시 더 작은 스토리로 분해한다.
> 4. Acceptance Criteria는 자동화된 테스트로 검증 가능하게 작성한다.
> 5. 의존성이 있는 경우 반드시 명시하고, 선행 작업의 완료 여부를 추적한다.
