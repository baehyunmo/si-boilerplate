## 2. 분류 기준

### 2.1 테스트 분류

| 분류 | 설명 |
|------|------|
| 정상 (Normal) | 정상적인 입력/흐름에 대한 테스트 |
| 예외 (Exception) | 비정상 입력, 오류 상황에 대한 테스트 |
| 경계값 (Boundary) | 입력값의 최소/최대/경계에 대한 테스트 |
| 성능 (Performance) | 응답시간, TPS, 동시 접속 등 성능 검증 |
| 보안 (Security) | 인증, 인가, 입력 검증 등 보안 검증 |
| 호환성 (Compatibility) | 브라우저, OS, 디바이스 호환성 검증 |

### 2.2 상태 정의

| 상태 | 설명 |
|------|------|
| 미실행 (Not Executed) | 아직 실행하지 않음 |
| 통과 (Pass) | 기대결과와 실제결과 일치 |
| 실패 (Fail) | 기대결과와 실제결과 불일치 → 결함 등록 |
| 차단 (Blocked) | 선행 조건 미충족으로 실행 불가 |
| 보류 (Deferred) | 현재 단계에서 실행 제외 |

### 2.3 심각도 기준

| 심각도 | 설명 |
|--------|------|
| Critical | 테스트 실패 시 시스템 운영 불가 |
| High | 테스트 실패 시 핵심 기능 장애 |
| Medium | 테스트 실패 시 부가 기능 장애 |
| Low | 테스트 실패 시 경미한 영향 |

> **작성 가이드**
> 1. TC ID는 모듈/기능별로 체계적으로 부여한다 (예: TC-USR-001, TC-ORD-001).
> 2. 전제조건은 테스트 재현에 필요한 모든 조건을 명시한다.
> 3. 테스트 단계는 누구나 따라할 수 있도록 구체적으로 작성한다.
> 4. 기대결과는 검증 가능한 형태로 작성한다 (화면 표시, 데이터 상태, 응답 코드 등).
> 5. 실패 시 반드시 결함을 등록하고 결함 ID를 연결한다.
> 6. 요구사항 추적 매트릭스(RTM)와 연결하여 커버리지를 관리한다.
