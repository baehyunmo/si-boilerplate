## 1. 개요 (Overview)

SI 프로젝트의 품질 보증을 위한 테스트 전략 문서이다.
TMMI(Test Maturity Model Integration) 프레임워크를 기반으로 레이어별 테스트를 정의하고,
자동화 전략과 품질 게이트 기준을 수립하여 체계적인 품질 관리를 수행한다.

### 1.1 목적
- 프로젝트 전 과정에 걸친 일관된 테스트 기준 수립
- 레이어별(Infra/Platform/Application) 테스트 전략 정의
- 자동화를 통한 테스트 효율성 및 반복성 확보
- 품질 게이트를 통한 단계별 품질 보증

### 1.2 적용 범위
본 전략은 다음 범위에 적용된다:
- 인프라 계층 (IaC, 네트워크, K8s)
- 플랫폼 계층 (CI/CD, 관측성, 정책)
- 애플리케이션 계층 (도메인 로직, API, UI)
- 거버넌스 계층 (문서, 프로세스)

## 2. TMMI 기반 테스트 성숙도

### 2.1 현재 성숙도 수준 및 목표

| TMMI 레벨 | 설명 | 현재 | 목표 |
|-----------|------|------|------|
| Level 1 | 초기 (Initial) | - | - |
| Level 2 | 관리 (Managed) — 테스트 정책/계획 수립 | 현재 | - |
| Level 3 | 정의 (Defined) — 조직 표준 테스트 프로세스 | - | 목표 |
| Level 4 | 측정 (Measured) — 정량적 품질 관리 | - | - |
| Level 5 | 최적화 (Optimized) — 지속적 개선 | - | - |

### 2.2 성숙도 향상 로드맵

| 단계 | 활동 | 산출물 | 일정 |
|------|------|--------|------|
| 1단계 | 테스트 정책/전략 수립 | 테스트 전략서 (본 문서) | 분석/설계 단계 |
| 2단계 | 테스트 프로세스 표준화 | 테스트 케이스 템플릿, 결함 관리 프로세스 | 설계/개발 단계 |
| 3단계 | 자동화 체계 구축 | CI/CD 파이프라인 내 테스트 자동화 | 개발 단계 |
| 4단계 | 정량적 측정 도입 | 테스트 대시보드, 메트릭 리포트 | 테스트 단계 |

## 3. 테스트 레벨 (Test Levels)

### 3.1 단위 테스트 (Unit Test)

| 항목 | 내용 |
|------|------|
| **대상** | Domain Layer, Application Layer (비즈니스 로직) |
| **도구** | Jest (TypeScript), pytest (Python), JUnit 5 (Java) |
| **커버리지 목표** | 라인 커버리지 80% 이상, 분기 커버리지 70% 이상 |
| **실행 시점** | 매 커밋 (pre-commit hook + CI) |
| **작성 책임** | 개발 에이전트 (app-agent) |
| **실행 시간 제한** | 전체 수행 5분 이내 |

**테스트 원칙:**
- 외부 의존성(DB, API, 파일시스템)은 Mock/Stub 처리
- 하나의 테스트는 하나의 행위(behavior)만 검증
- Given-When-Then 패턴 적용
- 테스트 이름은 `[테스트대상]_[시나리오]_[기대결과]` 형식

### 3.2 통합 테스트 (Integration Test)

| 항목 | 내용 |
|------|------|
| **대상** | Infrastructure Layer, Repository, 외부 시스템 연동 |
| **도구** | Testcontainers, Docker Compose |
| **커버리지 목표** | 주요 경로(Happy Path) 100%, 주요 예외 경로 80% |
| **실행 시점** | PR 생성 시 (CI 파이프라인) |
| **작성 책임** | 개발 에이전트 (app-agent) |
| **실행 시간 제한** | 전체 수행 15분 이내 |

**테스트 범위:**
- DB 연동: 실제 DB 컨테이너를 활용한 Repository 테스트
- API 연동: WireMock/MockServer를 활용한 외부 API 연동 테스트
- 메시지 브로커: 내장 Kafka를 활용한 이벤트 발행/소비 테스트
- 캐시: 실제 Redis 컨테이너를 활용한 캐시 테스트

### 3.3 시스템 테스트 (System Test / E2E Test)

| 항목 | 내용 |
|------|------|
| **대상** | 전체 시스템 (Interface → Domain → Infrastructure) |
| **도구** | Playwright (UI), k6 (성능), REST Assured (API) |
| **커버리지 목표** | 핵심 비즈니스 시나리오 100% |
| **실행 시점** | 스테이징 환경 배포 후 |
| **작성 책임** | QA 담당 / gov-agent |
| **실행 시간 제한** | 전체 수행 30분 이내 |

**테스트 유형:**
- API E2E 테스트: 전체 API 흐름 검증
- UI E2E 테스트: 핵심 사용자 시나리오 검증
- 성능 테스트: 부하/스트레스/내구성 테스트
- 보안 테스트: OWASP ZAP 자동 스캔

### 3.4 인수 테스트 (Acceptance Test / UAT)

| 항목 | 내용 |
|------|------|
| **대상** | 사용자 관점의 비즈니스 시나리오 |
| **도구** | 수동 테스트 + Playwright (자동화 가능 시나리오) |
| **수행 주체** | 현업 담당자 (고객) |
| **기준** | 요구사항 정의서의 수용 기준(Acceptance Criteria) 충족 |
| **실행 시점** | UAT 단계 (스테이징 환경) |
| **합격 기준** | 필수 요구사항 100% 통과, 선택 요구사항 90% 통과 |

| 서비스 메시 테스트 | Istio 설정 | istioctl analyze | mTLS, 라우팅 검증 |

### 4.3 애플리케이션 레이어 (Application Tier)

## 5. 자동화 전략

### 5.1 테스트 자동화 피라미드

```
        /  UAT  \          ← 수동 (핵심 시나리오만)
       /  E2E    \         ← 자동화 (핵심 경로)
      / 통합 테스트 \       ← 자동화 (주요 경로)
     / 단위 테스트    \     ← 완전 자동화 (최대 커버리지)
    ──────────────────
```

### 5.2 자동화 목표

| 테스트 레벨 | 자동화 비율 목표 | 실행 빈도 | 피드백 시간 |
|-------------|-----------------|-----------|-------------|
| 단위 테스트 | 100% | 매 커밋 | < 5분 |
| 통합 테스트 | 100% | PR 생성 시 | < 15분 |
| E2E 테스트 | 80% (핵심 시나리오) | 배포 후 | < 30분 |
| 성능 테스트 | 100% (자동 실행) | 릴리스 전 | < 1시간 |
| 보안 테스트 | 100% (SAST/DAST) | PR + 릴리스 전 | < 20분 |

### 5.3 CI/CD 파이프라인 내 테스트 통합

```
[코드 커밋]
    │
    ▼
[Pre-commit] ── lint, format, 단위 테스트
    │
    ▼
[PR 생성] ── 단위 테스트 + 통합 테스트 + SAST + 계약 테스트
    │
    ▼
[PR 머지] ── 빌드 + 이미지 스캔
    │
    ▼
[스테이징 배포] ── E2E 테스트 + DAST + 스모크 테스트
    │
    ▼
[운영 배포 승인] ── (수동 승인)
    │
    ▼
[운영 배포] ── 스모크 테스트 + 카나리 모니터링
```

## 6. 테스트 환경

| 환경 | 용도 | 데이터 | 인프라 규모 | 관리 주체 |
|------|------|--------|-------------|-----------|
| local | 개발자 로컬 테스트 | Mock / Fixture | Docker Compose | 개발자 |
| dev | 통합 개발 환경 | Seed Data | K8s (소규모) | infra-agent |
| staging | E2E / UAT / 성능 테스트 | 익명화된 운영 데이터 | K8s (운영 유사) | infra-agent |
| production | 스모크 테스트만 | 운영 데이터 | K8s (운영) | infra-agent |

### 6.1 테스트 데이터 관리

| 전략 | 환경 | 도구 | 비고 |
|------|------|------|------|
| Fixture/Factory | local, dev | Faker, Factory 패턴 | 코드 내 관리 |
| Seed Data | dev | 마이그레이션 스크립트 | 초기 데이터 |
| 데이터 익명화 | staging | 자체 스크립트 | 개인정보 마스킹 |
| 스냅샷 복원 | staging | DB 스냅샷 | 테스트 전 복원 |

### 6.2 테스트 환경 격리

- 각 PR 별 독립 네임스페이스 생성 (Ephemeral Environment)
- 테스트 완료 후 자동 정리 (TTL 기반)
- 스테이징 환경은 운영 환경과 동일한 설정 유지

### 7.1 결함 생명주기

```
[신규] → [할당됨] → [수정중] → [수정완료] → [검증중] → [종료]
                        │                        │
                        └──── [반려] ◄───────────┘
                                │
                                └──── [재오픈]
```

### 7.2 결함 심각도 기준

| 심각도 | 설명 | 수정 기한 | 예시 |
|--------|------|-----------|------|
| Critical | 시스템 운영 불가, 데이터 손실 | 즉시 (4시간 이내) | 서비스 다운, 데이터 유실 |
| Major | 핵심 기능 장애, 우회 불가 | 1 영업일 이내 | 주요 업무 기능 오류 |
| Minor | 기능 장애, 우회 가능 | 해당 스프린트 이내 | 비핵심 기능 오류 |
| Trivial | 경미한 UI/문구 오류 | 다음 릴리스 이전 | 오타, 디자인 미세 조정 |

|------|------|
| 결함 ID | BUG-XXXX |

| 심각도 | Critical / Major / Minor / Trivial |
| 재현 절차 | 결함 재현을 위한 단계별 절차 |
| 기대 결과 | 정상 동작 시 기대 결과 |
| 실제 결과 | 실제 발생한 결과 |

| 스크린샷/로그 | 증빙 자료 |
| 담당자 | 수정 담당자 |
| 연관 TC | 관련 테스트 케이스 ID |

## 8. 품질 게이트 기준 (Quality Gates)

### 8.1 단계별 품질 게이트

#### Gate 1: PR 머지 기준

- [ ] 단위 테스트 커버리지 80% 이상 (신규 코드)
- [ ] 단위 테스트 전체 통과
- [ ] 통합 테스트 전체 통과
- [ ] SAST 스캔 — Critical/High 취약점 0건
- [ ] 코드 리뷰 승인 1명 이상
- [ ] 린트/포맷 규칙 준수

#### Gate 2: 스테이징 배포 기준

- [ ] Gate 1 조건 충족
- [ ] E2E 핵심 시나리오 전체 통과
- [ ] DAST 스캔 — Critical/High 취약점 0건
- [ ] 컨테이너 이미지 스캔 — Critical 취약점 0건
- [ ] 성능 테스트 기준 충족 (응답시간, TPS)

#### Gate 3: 운영 배포 기준

- [ ] Gate 2 조건 충족
- [ ] UAT 합격 (필수 요구사항 100%)
- [ ] 보안 점검 완료
- [ ] 운영 팀 인수 승인
- [ ] 롤백 계획 수립 완료
- [ ] 모니터링/알림 설정 완료

#### Gate 4: 안정화 완료 기준

- [ ] 운영 2주간 Critical/Major 결함 0건
- [ ] SLA 기준 충족 (가용률, 응답시간)
- [ ] 잔여 결함 Minor 이하로 종결/이관
- [ ] 운영 매뉴얼 최종 갱신 완료

### 8.2 품질 메트릭 대시보드

| 메트릭 | 측정 방법 | 목표 | 현재 |
|--------|-----------|------|------|
| 테스트 커버리지 (단위) | CI 파이프라인 | 80% | - |
| 테스트 커버리지 (통합) | CI 파이프라인 | 주요 경로 100% | - |
| 결함 밀도 | 결함 수 / KLOC | < 5건/KLOC | - |
| 결함 탈출률 | 운영 결함 / 전체 결함 | < 5% | - |
| 테스트 자동화율 | 자동화 TC / 전체 TC | > 80% | - |
| 평균 결함 수정 시간 | 결함 등록 ~ 종료 시간 | < 2 영업일 | - |
| 회귀 결함율 | 회귀 결함 수 / 전체 결함 | < 3% | - |
